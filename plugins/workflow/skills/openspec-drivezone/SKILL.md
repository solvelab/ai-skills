# OpenSpec — the DriveZone rite (forked schema)

> **What this is.** DriveZone uses [OpenSpec](https://github.com/Fission-AI/OpenSpec) as its
> spec-driven process — the base workflow is the **`openspec` skill** — but with a **project-local
> forked schema** that makes three sections **mandatory** where vanilla has none: **Fallback & Failure
> Modes**, **Tests & Bug-Hunter**, and **Validation & Closure**.
>
> Two parts: **Part 1 — For the AI** (dense, imperative: what to execute) and **Part 2 — For humans**
> (why each rule exists, and the difference vs vanilla).

---

## Part 1 — For the AI (execute)

### Where everything lives

Each active subproject has its own `openspec/` — **there is no monorepo-global OpenSpec**.

| Subproject | OpenSpec path | Test track |
|---|---|---|
| `fivem-drivezone/` (Lua/FiveM) | `fivem-drivezone/openspec/` | busted (pure modules) + fallback at the caller |
| `backend-drivezone/` (FastAPI) | `backend-drivezone/openspec/` | pytest E2E via TestClient |

Internal layout, lifecycle commands and artifact formats: standard OpenSpec — see the
**[openspec]** skill. The fork adds `schemas/spec-driven/schema.yaml` + templates that force the
three gates below. The `/opsx` skills are **scoped per subproject** and generated by the CLI
(`openspec update`) — **never hand-edit them**.

### The three mandatory gates (the differentiator)

**1. `design.md` → section `## Fallback & Failure Modes (MANDATORY)`**

A 3-column table. For **every** external dependency, declare the failure mode + safe default. Where
there is no fallback, **justify it**.

```markdown
| Dependency | Failure mode | Safe default / fallback |
|---|---|---|
| <dep> | <timeout / 5xx / down / partial payload / race / disconnect> | <safe default, no crash, no silent stale state> |
```

Dependencies to cover per track:
- **FiveM**: `city-rest-api`, Consul `/config/*`, network, another resource's event, NUI callback.
  → Mechanics: **[fivem-fallback]** (Lua adaptation of **[backend-resilience]**).
- **Backend**: Consul `/config/*`, integrations, DB/transaction.
  → Mechanics: **[backend-resilience]**; error envelope `status:error` + `RC_*` (never a raw 500).

**2. `tasks.md` → group `## 3. Tests & Bug-Hunter (MANDATORY)`**

Adversarial QA, not happy-path. → Methodology and per-stack scenarios: **[bug-hunter]**
(`references/track-fivem-lua.md` for the resource, `references/track-python-pytest.md` for the
backend — pytest E2E runs **before** any in-game validation). For API-surface depth: **[api-resilience-testing]**.
Trust-boundary rules exercised by the Lua track: **[fivem-lua]**.

**3. `tasks.md` → group `## 4. Validation & Closure (MANDATORY)`** (last)

- FiveM: `openspec validate <id> --strict` green, **documented in-game smoke**, resource README +
  `KEYBINDS.md` updated, `openspec archive <id> --yes`.
- Backend: `openspec validate <id> --strict` green, **pytest green**, `openspec archive <id> --yes`.

### Execution rules (imperative summary)

- **Always** run `openspec validate <id> --strict` **before** implementing (it is the gate).
- **Backend**: adversarial pytest E2E **before** in-game validation (the AI doesn't run the game).
- **FiveM**: testable pure modules (busted) + fallback at the caller + the hunt — never happy-path only.
- **Never** hand-edit the generated `/opsx` skills or `schema.yaml`; they change via CLI / deliberate fork.
- Current specs live in `openspec/specs/<capability>/spec.md`; `archive/` is history and delta base.

## Part 2 — For humans (the why, and the diff vs vanilla)

### Vanilla OpenSpec

Standard OpenSpec gives the spec-driven flow: proposal (Why/What), spec deltas (ADDED/MODIFIED/REMOVED
requirements with WHEN/THEN scenarios), tasks, and archive syncing the specs. Templates are generic —
they fit any project. Full reference: the **[openspec]** skill.

### What the DriveZone fork adds

The fork doesn't change the *flow* — it changes the **mandatory templates** via a project-local
`schema.yaml`. It turns three disciplines that used to be "good practice we remember to do" into
**gates `validate --strict` refuses to skip**. Rigor stops depending on the dev's (or the AI's)
memory and becomes **structural**.

**Why `Fallback & Failure Modes` is mandatory.**
DriveZone is distributed by nature: a Lua resource talks to a FastAPI backend, which talks to Consul,
in a game where the player can disconnect mid-flow. Without declaring each dependency's failure mode
**at design time**, code is born assuming the happy path and breaks in production. Forcing the table
into `design.md` makes resilience a design input, not a patch. Execution mechanics live in
**[backend-resilience]** / **[fivem-fallback]**.

**Why `Tests & Bug-Hunter` is mandatory.**
Project premise: **every feature/fix requires adversarial QA** before in-game validation — the AI
cannot play the game, so the backend must be proven by adversarial pytest E2E and the Lua by testable
pure modules + the hunt. The *how* lives in **[bug-hunter]** and **[api-resilience-testing]**.

**Why `Validation & Closure` is mandatory and final.**
It closes the loop: `validate --strict` green + proof (pytest green / documented in-game smoke) +
docs updated + `archive`. Without this group, "done" is an opinion; with it, "done" is verifiable.

### FiveM vs Backend — mirrored schemas, different content

Both subprojects share the **same gate skeleton**, but each schema names the dependencies and tests of
**its** reality:

| | FiveM (`fivem-drivezone`) | Backend (`backend-drivezone`) |
|---|---|---|
| Fallback deps | city-rest-api, Consul, network, resource events, NUI callback | Consul, integrations, DB/transaction |
| Test track | busted (pure modules), fallback at caller, concurrent StateBags, disconnect cleanup | pytest E2E (TestClient), `FOR UPDATE`, negative cache, atomicity/idempotency |
| Closure proof | documented in-game smoke + README/KEYBINDS | pytest green |

### Where this skill fits in the ecosystem

This skill is the **lifecycle orchestrator**. Each discipline it makes mandatory has its own skill
describing the execution:

- **[openspec]** — the vanilla workflow this rite extends.
- **[backend-resilience]** — stack-agnostic resilience doctrine → the *Fallback* gate.
- **[fivem-fallback]** — Lua adaptation → the *Fallback* gate (FiveM track).
- **[bug-hunter]** — adversarial QA rite → the *Tests & Bug-Hunter* gate.
- **[api-resilience-testing]** — API negative/fuzz/contract testing → reinforces the backend test gate.
- **[fivem-lua]** — Lua resource conventions (trust boundary, fxmanifest, NUI) → implementation base.
